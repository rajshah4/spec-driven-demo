Why Not "Vibe Coding"?

When you ask AI to "just build it," you get unpredictable results

😵 Vibe Coding

  • "Add photo sharing to my app"
  • AI guesses flow, tech stack, integration
  • Unpredictable, inconsistent results
  • "Nobody likes to vibe debug"

✅ Spec-Driven

  • Structured spec → Plan → Tasks → Code
  • You define what and why, AI handles how
  • Production-ready, consistent outcomes
  • Systematic, reviewable process

What is OpenSpec?

A structured, machine-readable format that makes specs actionable — not just readable

📐 Schema-Enforced

Every spec.json is validated against a JSON Schema. Required fields — user_stories, acceptance_criteria, requirements — must be present before the workflow proceeds.

🤖 AI-Ready

Structured JSON means agents can parse intent reliably. No ambiguity — must-have requirements are separate from nice-to-haves, out-of-scope is explicit.

Why JSON, not Markdown?

// Validated automatically in CI
{
  "version": "1.0",
  "user_stories": [...],        ✓ required
  "requirements": {
    "must_have": [...],         ✓ required
    "should_have": [...],
    "nice_to_have": [...]
  },
  "acceptance_criteria": [...], ✓ required
  "out_of_scope": [...]         ✓ explicit
}

Schema lives at .specify/schema/spec.schema.json — versioned in your repo.

Step 1: SPECIFY

Triggered when you create a GitHub Issue

What Happens

  • Agent reads your issue description
  • Explores the codebase for context
  • May ask clarifying questions
  • Creates structured spec.json
spec-ready ← Added when done

Output: spec.json

{
  "version": "1.0",
  "feature": "User Authentication",
  "problem_statement": "Users need secure access...",
  "user_stories": [...],
  "requirements": {
    "must_have": [
      "Email/password login",
      "Session management"
    ]
  },
  "acceptance_criteria": [
    "Login succeeds with valid credentials",
    "Error returned on invalid credentials"
  ]
}
🔧

Step 2: PLAN

Triggered when you add spec-approved label

What Happens

  • Agent reads approved spec
  • Analyzes existing architecture
  • Defines tech stack & patterns
  • Creates plan.md with details
spec-approved ← You add this
plan-ready ← Agent adds when done

Output: plan.md

# Implementation Plan: User Auth

## Technology Stack
- **Framework:** Next.js 14
- **Auth:** NextAuth.js
- **Database:** PostgreSQL

## Architecture
### Components
1. AuthProvider - Session management
2. LoginForm - User interface

## File Changes
### New Files
- `lib/auth.ts` - Auth configuration
- `components/LoginForm.tsx`
📝

Step 3: TASKS

Triggered when you add plan-approved label

What Happens

  • Agent reads spec + plan
  • Breaks down into small tasks
  • Orders by dependencies
  • Marks parallel tasks with [P]
plan-approved ← You add this
tasks-ready ← Agent adds when done

Output: tasks.md

# Task Breakdown: User Auth

## Phase 1: Foundation
### Task 1.1: Create auth config
- **File:** `lib/auth.ts`
- **Depends on:** None

### Task 1.2: Create user model [P]
- **File:** `prisma/schema.prisma`
- **Parallel:** Yes (with 1.3)

## Phase 2: Core Implementation
### Task 2.1: Build LoginForm
- **File:** `components/LoginForm.tsx`
- **Depends on:** Task 1.1
⚡

Step 4: IMPLEMENT

Triggered when you add ready-to-implement label

What Happens

  • Agent creates feature branch
  • Executes tasks in order
  • Makes atomic commits per task
  • Runs tests before PR
  • Opens Draft PR
ready-to-implement ← You add this

Output: Draft Pull Request

● feat: User Authentication (#42)
Closes #42
Changes:
✅ Task 1.1: Create auth config
✅ Task 1.2: Create user model
✅ Task 2.1: Build LoginForm
✅ Task 3.1: Add unit tests
💬

Step 5: REFINE

Triggered when you submit a PR review

What Happens

  • Agent reads review comments
  • Evaluates each piece of feedback
  • Makes necessary changes
  • Responds to comment threads
  • Marks PR ready when addressed
💡 Smart Evaluation

Agent prioritizes bug fixes and security issues, but can respectfully decline over-engineering suggestions.

Review Response

👤 Reviewer:
"Should add input validation here"
🤖 Agent:
"Good catch! Added validation in commit abc1234. Also added error messages for invalid inputs."
